Multi-Agent Banking Assistant – Agentic AI Solution

Project Overview

The Multi-Agent Banking Assistant is a production-ready Agentic AI solution designed to streamline loan inquiries, compliance checks, and knowledge retrieval within a banking environment. It leverages modular agent design, asynchronous execution, and retrieval-augmented generation (RAG) principles to deliver intelligent, reliable, and scalable responses. The system integrates structured transactional data with a knowledge base and ensures regulatory compliance through dedicated agents.

Its primary goal is to enhance customer experience, automate repetitive inquiries, and enforce bank policies consistently, while remaining adaptable for future integration with large language models such as Titan or Claude.

System Architecture

The system uses a multi-agent architecture, with each agent responsible for a specific function. Agents communicate asynchronously, ensuring high throughput and low latency. The primary components are:

  • DBManager: Handles async CRUD operations, JSON population, and FTS5 full-text search.
  • ComplianceAgent: Evaluates customer requests against banking policies.
  • GuardrailsAgent: Applies business and safety rules to filter risky requests.
  • CanonicalAgent: Maps inputs to canonical forms.
  • IntentAgent: Extracts intent and relevant slots from user queries.
  • RAGAgent: Retrieves knowledge base information for LLM supplementation.
  • ResponseAgent: Combines outputs to generate final responses.

Database Management (DBManager)

The DBManager provides thread-safe access to multiple tables:

  • Contacts Table: Customer details (name, email, phone).
  • Loans Table: Customer loan info (type, status, disapproval reasons).
  • Compliance Table: Banking policies and regulatory rules.
  • Policy Table: Internal operational policies.
  • FTS5 Knowledge Base Table: Full-text search for knowledge retrieval.

Features: Async and thread-safe with caching, JSON population, RAG-ready design. Ensures atomic, consistent, performant operations.

ComplianceAgent

  • Policy Verification: Checks if requests violate regulations.
  • Rule Enforcement: Loan limits, KYC, AML checks.
  • Fallback to KB: Queries RAG system if explicit rules are insufficient.

Runs asynchronously, outputs structured data, and is ready for LLM integration.

GuardrailsAgent

  • Validates requests against internal policies.
  • Provides safe defaults or rejects risky inputs.
  • Prepares system for LLM-based reasoning in the future.

CanonicalAgent

Standardizes user input into canonical forms.

{
        "intent": "get_loan_status",
        "customer_id": 1,
        "loan_type": "personal",
        "loan_id": "987654"
    }

IntentAgent

  • Extracts purpose of queries (loan status, policy info, contact updates).
  • Slot filling (loan ID, customer ID, loan type).
  • Regex and rules-based approach, future LLM integration.

RAGAgent

  • FTS5 full-text search for relevant knowledge retrieval.
  • Fallback templates ensure coherent answers.
  • Ready for vector-based semantic search.
  • Async operation prevents bottlenecks.

ResponseAgent

  • Integrates outputs from all agents.
  • Formats human-readable messages.
  • Supports template responses and future LLM generation.
  • Error handling with safe fallback responses.

Main Application Workflow

  1. User queries loan status.
  2. IntentAgent extracts intent and slots.
  3. CanonicalAgent normalizes input.
  4. ComplianceAgent checks rules.
  5. GuardrailsAgent enforces safety.
  6. RAGAgent retrieves contextual knowledge.
  7. ResponseAgent composes final response.
